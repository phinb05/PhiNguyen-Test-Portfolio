Cold Email Workflow

The Pain

Generic cold emails get ignored in one swipe. Hand-writing personalised ones tops out at ~20 per morning, and tracking who's been emailed is how double-sends and spam hits happen.

The Build

An n8n loop reads leads from Google Sheets, builds a per-lead Claude Sonnet prompt with the company profile, sends via Gmail, and writes sent back to the row — all resumable mid-run.

The Result

100%

every email personalised at scale

Architecture

The pipeline

  • Read leads from a Google Sheet whose schema matches the upstream Lead Generation system: name, email, company, website, description, keywords, plus a send_status column.
  • Loop in batches so a failure on one row doesn't take the whole run down.
  • Skip if already sent — the conditional on send_status is what makes the workflow resumable. Stop it mid-run, start it again, nobody gets double-emailed.
  • Build the prompt in a Code node from the lead's company profile plus sender details factored out as constants at the top.
  • Generate with Claude Sonnet via a direct Anthropic Messages API call. The system prompt locks tone ("warm and genuinely curious, never salesy"), length (<130 words), and output format (strict JSON: subject + body).
  • Parse & validate — strips markdown fences, throws with the raw response attached on failure so debugging isn't a guessing game.
  • Send via Gmail, then mark the row as sent — which makes the skip-check on the next run do its job.

Design decisions worth calling out

  • Safe to stop and restart. Send-state lives in the same sheet the workflow reads from. The two halves can never drift apart, so interrupting a run is a non-event.
  • Strict JSON output with a safety net. LLMs that "almost" return valid JSON are a common production breakage. The prompt forbids fences, the parser strips them anyway.
  • Sender identity factored out of the template. Switching from "Alex at Acme" to your own name/company is editing three constants, not rewriting a prompt.

Why this matters

This is the outreach half of a complete outbound pipeline. It consumes the exact schema the Lead Generation workflow produces, so the two compose into a clean handoff: discover & enrich → personalise & deliver, with send-state as the only shared contract.

Multi-Channel Customer Support Bot

The Pain

A Vietnamese preschool couldn't staff live chat on Messenger, Telegram, and the website widget at once. Every minute of delay was a parent drifting to a competitor.

The Build

One LangChain agent (GPT-4o-mini via OpenRouter) sits behind three channel adapters. Shared knowledge base, per-session memory, Vietnamese phone regex for automatic lead capture.

The Result

3-in-1

live channels, one source of truth, deployed to a real client

Architecture

One brain, three faces

A single n8n workflow runs one shared AI agent behind three channel adapters. Knowledge, tone rules, and escalation logic live in exactly one place. Adding a channel means adding an adapter — not rebuilding the whole thing.

  • AI Consultant agent — LangChain Agent + GPT-4o-mini carrying the full Vietnamese knowledge base for Sakuranbo Kindergarten: pricing tiers by age, schedules, admission, meals, discipline, extracurriculars, transport, dietary rules. System prompt locks tone (Vietnamese honorifics, "nhà trường", "anh/chị"; no casual pronouns) and factual boundaries (never invent prices — defer to staff when unsure).
  • Telegram adapter — Telegram Trigger → Extract Content → Detect Phone → route.
  • Facebook Messenger adapter — FB webhook (Verify + Message) → Extract FB Data → Get FB Profile → Detect Phone → route.
  • Website chat adapter — Web Chat Trigger → Web Detect Phone → route.
  • Per-session memory — LangChain Buffer Window memory keyed by sender ID, so each parent's thread is isolated.

Automatic lead capture

A Vietnamese phone regex (0[35789]\d{8}) runs on every incoming message. When a parent shares a number, the message is flagged, logged, and a separate Telegram notification fires to the staff's internal channel with name and phone. The agent is prompted to proactively ask for a number after two or three exchanges or on intent signals ("how do I register", "when can my child start"). Passive chat traffic becomes qualified leads without any manual triage.

Production details that usually get skipped

  • Prompt injection defense. A "TUYỆT MẬT" section in the system prompt refuses to reveal its own contents, blocks roleplay attacks ("pretend you are X", "ignore previous instructions"), and refuses internals questions — routing back to a safe canned response.
  • PII hashing. Sender IDs are SHA-256 hashed before being written to the log, so analytics stays useful without a direct identifier back to individual users.
  • Unified logging. All three channels write to the same Google Sheet with timestamp, hashed sender, question, response, and channel — so conversation history is searchable in one place.
  • Error trigger + staff alerting. A dedicated Error Trigger catches any failure and fires a Telegram alert with failing node, error message, and execution ID. Silent failures are the worst class of bug in customer-facing automation.

The decision worth calling out

The "one brain, three faces" split is what makes this maintainable. When the school raises a fee, I update one number in one system prompt and all three channels are immediately accurate. When Messenger breaks its webhook format, I patch one adapter and the other two surfaces keep running. Shared intelligence, isolated plumbing — that separation is the single most important thing this project taught me.
